# BMS Data Logger

## Project description
This repository contains a C++20 Battery-Management-System (BMS) data-logger for battery packs. The application polls two Modbus/TCP voltage devices and one Modbus/TCP temperature device, then writes the resulting samples to InfluxDB 3 using Line Protocol over HTTP. The runtime architecture follows a real-time producer/consumer design with lock-free queues and preallocated memory pools to minimize allocation jitter and dropped data under load.

Key dependencies include:
- Boost (`system`, `thread`, `chrono`, lock-free/atomics)
- Eclipse Paho MQTT
- libmodbus
- libcurl
- nlohmann/json

The repository also includes helper scripts to bootstrap InfluxDB access and schema:
- `scripts/get_token.sh` to request and save an API token to `config/influxdb3/token.json`
- `scripts/setup_schema.sh` to create voltage/temperature tables

## Directory structure
- `app/inc` – C++ headers for core types and modules (`batch_structures.hpp`, pools, queues, Modbus client, voltage/temperature acquisition, InfluxDB interfaces).
- `app/src` – C++ implementation files (`main.cpp`, `modbus_reader.cpp`, `influxdb.cpp`) containing runtime logic.
- `config/influxdb3` – persistent InfluxDB data and `token.json` generated by `scripts/get_token.sh`.
- `config/mosquitto` – Eclipse Mosquitto configuration files.
- `scripts` – bootstrap/operations scripts such as `get_token.sh`, `setup_schema.sh`, and Raspberry Pi setup helpers.
- `cmake` – CMake modules including `toolchain-armhf.cmake` for ARM cross-compilation.
- `utils` – utility resources, patterns, dashboards and support files.
- `rasp` – Raspberry Pi deployment assets (`Dockerfile`, `docker-compose.yml`, `entrypoint.sh`).
- `.devcontainer` – VS Code dev-container configuration for reproducible development.

## Local host setup
### Prerequisites
- C++20-compatible compiler (GCC/Clang)
- CMake 3.22+
- Boost
- libmodbus
- libcurl
- Paho MQTT C/C++ libraries
- nlohmann/json
- Docker Engine (for service containers)
- `jq` (required by shell scripts)

### Build
```bash
mkdir build && cd build
cmake ..            # optionally: -DCMAKE_BUILD_TYPE=Release
cmake --build . --target bms
```

The executable is generated at `bin/bms` (relative to the build output layout).

### Start support services with Docker Compose
From repository root:
```bash
docker compose -f rasp/docker-compose.yml up --build -d
```

This compose stack launches:
- `bms`
- `mosquitto`
- `influxdb3`

After InfluxDB is up, run:
```bash
scripts/get_token.sh
scripts/setup_schema.sh
```

These commands generate `config/influxdb3/token.json` and create the voltage/temperature tables used by the logger.

### Run directly on host (without bms container)
1. Start InfluxDB 3 and Mosquitto manually (e.g., official Docker images).
2. Set host/port/credentials in `config/influxdb3/token.json`.
3. Run:
   ```bash
   ./bin/bms
   ```
4. Ensure `jq` is installed so helper scripts function correctly.

## Raspberry Pi deployment
### Option 1: Cross-compile and run natively
Use the provided ARM toolchain file:
```bash
mkdir build-arm && cd build-arm
cmake .. -DCMAKE_TOOLCHAIN_FILE=cmake/toolchain-armhf.cmake
cmake --build . --target bms
scp bin/bms pi@raspberrypi:/home/pi/bms/
# On the Pi (with dependencies installed):
./bms
```

On the Raspberry Pi, ensure runtime libraries are installed (`libboost`, `libmodbus`, `libcurl`, etc.) and that the device can reach Modbus devices and InfluxDB over the network.

### Option 2: Containerized deployment on Pi
Copy the `rasp/` content to the Pi and run:
```bash
docker compose up --build -d
```

This will build the Pi image, start InfluxDB and Mosquitto, execute `entrypoint.sh` (token/schema bootstrap), and then start the BMS application automatically.

Monitor startup with:
```bash
docker compose logs -f
```

If preparing a fresh Raspberry Pi OS install, use:
```bash
scripts/rasp-pre_setup.sh
```

## Usage and shutdown
- Run the logger and keep it active while data acquisition is needed.
- Stop with `Ctrl+C`; the application handles shutdown gracefully by stopping periodic tasks, joining threads, and disconnecting services.
- Every 10 seconds the app prints diagnostics (published/dropped samples, queue/pool stats, and InfluxDB write statistics).
